## Introduction

Webhooks let you receive real-time notifications for Dots events such as Payout Links, Flows, Transfers, Users, Apps, and Organizations. Dots sends webhook events as POST requests to the endpoint you configure in the dashboard.

Return any 2xx status code within 15 seconds to acknowledge a webhook. If your framework enables CSRF protection by default, disable CSRF protection for the webhook endpoint.

Every webhook payload includes an `event` field whose value matches the event type. App-scoped events delivered to an organization-level endpoint can also include `api_app_id`, which identifies the app that originated the event.

## Configuring Webhooks

Dots uses [Svix](https://docs.svix.com/receiving/introduction/) to deliver webhooks.

1. Open the [Dots dashboard](https://dashboard.dots.dev/).
2. Select an App, or open Organization settings for organization-level webhooks.
3. Go to **API Management** and open the Svix dashboard link.
4. Click **+ Add Endpoint**.
5. Enter your endpoint URL.
6. Select the event types the endpoint should receive. If no event is selected, Svix sends all event types.
7. Click **Create**.
8. Save the webhook signing secret.

## Event Catalog

The webhook payload schema is generated from the same backend event registry used to validate outbound payloads. Use the **Webhook Events** pages in the sidebar to view rendered payload schemas for each event type.

The machine-readable OpenAPI webhook catalog is also available at [`/webhooks.yaml`](/webhooks.yaml).

Available event types:

| Event | Description |
| --- | --- |
| `app.updated` | An app object was updated. |
| `dispute.updated` | A dispute was updated. |
| `flow.updated` | A flow was updated after the user completed additional steps. |
| `organization.updated` | An organization object was updated. |
| `payment_intent.updated` | A payment intent object was updated. |
| `payment_method.updated` | A payment method object was updated. |
| `payout_link.claimed` | A payout link was claimed by a user. |
| `payout_link.failed` | A payout link funding transfer failed. |
| `payout_link.paid_out` | A user completed a payout through a payout link. |
| `payout_link.sent` | A payout link was sent by email, SMS, or direct link. |
| `payout_link.updated` | A payout link was updated. |
| `payout_request.updated` | A payout request changed status. |
| `transfer.completed` | A transfer completed. The payload includes the transfer object. |
| `transfer.failed` | A transfer failed. The payload includes the transfer object. |
| `transfer.flagged` | A transfer was flagged. The payload includes the transfer object. |
| `transfer.updated` | A transfer was updated. |
| `user.profile_changed_external` | An external payout provider reported user profile details that differ from Dots. |
| `user.updated` | A user was updated. |
| `user.updated.background_check` | A user's background check was updated. |
| `user.updated.default_payout_method_changed` | A user's default payout method changed. |
| `user.updated.id_verified` | A user's identity verification completed. |
| `user.updated.manual_credit` | A manual credit transfer was recorded for a user. |
| `user.updated.manual_debit` | A manual debit transfer was recorded for a user. |
| `user.updated.payout_method` | A user's payout method was updated. |
| `user.updated.user_merged` | Two user records were merged. |

## Webhook Signing

Verify webhook signatures before processing events. See the [Svix signature verification guide](https://docs.svix.com/receiving/verifying-payloads/why).
